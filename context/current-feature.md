## Goals

- A user can sign up, sign in, play Event Bubbling Bubbles, and see their
  completion persist after a reload or on another device; unauthenticated
  users are prompted to sign in before progress-relevant actions are saved.
- Prisma schema for `User` (via Better Auth), `UserProgress`, `Attempt` on
  PostgreSQL, per the Core Data Models in `context/project-overview.md`.
- Better Auth configured with the Prisma adapter; email/password or
  magic-link sign-in for MVP (social providers out of scope).
- Real `/auth/sign-in` and `/auth/sign-up` routes replacing feature-03
  placeholders.
- Vercel Functions API layer (`api/` + `src/server`) exposing a session-aware
  progress read and an idempotent completion mutation (keyed by
  `clientAttemptId`).
- All API inputs/outputs validated with Zod; route handlers verify session
  ownership before reading/writing progress; strict layering (route handler
  → auth/session check → Zod validation → domain service → Prisma
  repository), no direct Prisma calls from route handlers.
- Wire Event Bubbling Bubbles' result flow (feature 05) to the real
  completion mutation instead of local-only state.
- Wire the Track/Lesson catalog views (feature 03) to reflect real
  `UserProgress.masteryState` for the signed-in user.

### Implementation decisions

- Sign-in is **email/password**. Magic-link needs a mail transport that is not
  provisioned; email verification is off for the same reason (turning it on
  without a transport would lock every new account out).
- Play is **not gated**. Sign-in is prompted on the result screen, at the point
  progress would be saved. The finished attempt is persisted to `sessionStorage`
  so it survives the sign-in round trip (and a reload) and still saves.
- Idempotency key (`clientAttemptId`) is generated by the runtime when an
  attempt finishes, so every future mini-game inherits it. The server relies on
  the `(userId, clientAttemptId)` unique index — not a pre-read — so concurrent
  replays cannot both slip through.
- Env vars are documented in `.env.example` and the README
  ("Environment and database"): `DATABASE_URL`, `BETTER_AUTH_SECRET`,
  `BETTER_AUTH_URL`.

### Follow-ups (deliberately not done here)

- **Rate limiting** — not added beyond Better Auth's defaults. Sign-in/sign-up
  and the completion mutation are unprotected against brute force/abuse.
- **Server-authoritative scoring** — score, round counts, and level mode are
  client-reported. Trustworthy scoring needs level definitions on the server.
- **Playwright E2E** (`e2e/auth-progress.spec.ts`, gated behind `E2E_WITH_API=1`)
  is written but has not been executed; the flow it covers was verified manually
  through the API instead.

### Verified against a live database

Migrations applied to Prisma Postgres and the full flow driven through the real
API (`npm run dev` now serves `/api/*` via `vite/apiDevServer.ts`):

- Sign up → session cookie issued → completion saved → progress read back.
- Mastery advanced `applied` → `mastered`; the original `completedAt` was kept
  rather than being moved by the later attempt.
- **Idempotency under real concurrency**: five parallel writes of the same
  `clientAttemptId` produced exactly one `attemptRecorded: true` and one row.
- **Ownership**: a second user reads an empty progress list, and a `userId`
  planted in the request body is ignored — the write lands on the session's
  user, leaving the other account untouched. Signed-out reads are 401.

Test data was removed afterwards; the database is empty.

**Bug found and fixed by doing this** — the unit tests could not have caught it.
The repository originally inserted the attempt and caught the unique-constraint
violation as its replay signal. In Postgres a failed statement aborts the whole
transaction, so the follow-up read that answers the replay died with `25P02` and
the API returned 500 on every retry. It now uses `createMany({ skipDuplicates })`
(`ON CONFLICT DO NOTHING`), which reports the conflict via the row count without
poisoning the transaction. The in-memory fake detected duplicates with a set
lookup, so it reported success while the real code was broken; that gap is now
covered by the opt-in integration tests.

### Event Bubbling Bubbles (First Mini-Game)

- Built the first real, shippable mini-game (`src/games/event-bubbling-bubbles`)
  against the feature 04 runtime contract with no forking: discover/apply/
  master levels teaching DOM event bubbling/propagation through a
  nested-bubble tap metaphor, with pure domain logic (`domain/bubbling.ts`)
  computing the reacting-layer set for a tap and comparing it against the
  learner's selection.
- Master level introduces stopPropagation-equivalent "stop" placement: the
  learner places a stop at a layer, then predicts the new reacting set — the
  stopped layer still reacts, nothing further outward does.
- GSAP timeline animates the bubble outward through reacting layers on the
  runtime's `simulating` stage, with a reduced-motion fallback that steps
  through the same sequence via staged highlights instead of continuous motion.
- Added the centralized audio service (`src/lib/audio/audioService.ts`) —
  a small Web Audio abstraction synthesizing short tones per cue rather than
  loading sample files — and a haptics service stub (`src/lib/haptics/
  hapticsService.ts`, no-op until Capacitor in feature 12). Both are globally
  disableable and never the sole feedback channel.
- Wired into `/play/event-bubbling-bubbles` and its result screen.
- Unit tests cover the domain logic and win/failure evaluation, plus a
  component test suite (correct/incorrect predictions, master stop
  placement, keyboard-only playthrough). Storybook stories cover the
  predicting, master stop-placement, transfer, and explaining game states,
  plus the shared `BubbleLayerList` visual in mid-propagation, success, and
  mistake states.
